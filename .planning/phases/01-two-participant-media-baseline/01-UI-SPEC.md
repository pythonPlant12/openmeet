# Phase 01 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn-vue |
| Preset | `new-york`; Tailwind v3; base color `zinc`; CSS variables enabled; prefix empty |
| Component library | Reka UI via existing shadcn-vue primitives |
| Icon library | lucide-vue-next |
| Font | Tailwind default system sans-serif |

**Source:** `openmeet-client/components.json`, `npx shadcn-vue@latest info`, `openmeet-client/src/components/ui/**`, and existing meeting page components.

**Phase posture:** Preserve the current dark meeting UI. This phase may extend existing status and error surfaces for reliability diagnostics, but it must not redesign the meeting experience or introduce new visual systems.

---

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badges, compact debug rows |
| sm | 8px | Control gaps, badge padding, dialog footer gaps |
| md | 16px | Default component spacing, video grid gaps, panel padding |
| lg | 24px | Dialog content padding, section padding |
| xl | 32px | Single-participant video padding, large layout gutters |
| 2xl | 48px | Circular meeting controls, major component separation |
| 3xl | 64px | Page-level separation only; avoid adding for this phase unless preserving existing layout |

Exceptions: circular meeting controls remain `48px × 48px`; join-dialog media toggle buttons remain `56px` high; all click/tap targets introduced or changed in this phase must be at least `44px × 44px`; video tiles may keep viewport-based heights and `aspect-video` sizing because media layout depends on available viewport.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `hsl(var(--background))` = `hsl(0 0% 12%)` / `#1f1f1f`; video canvas may use existing `hsl(0 0% 8%)` / `#141414` | Meeting page background, video tile base, full-screen loading states |
| Secondary (30%) | `hsl(var(--card))` = `hsl(0 0% 18%)` / `#2e2e2e`; `hsl(var(--muted))` = `hsl(0 0% 28%)` / `#474747` | Bottom controls bar, dialogs, debug/status panels, preview placeholders, secondary controls |
| Accent (10%) | `hsl(var(--primary))` / `hsl(var(--accent))` = `hsl(154 94% 40%)` / approx. `#06c668` | Primary CTA, active status button, participant count badge, local avatar, successful/connected status text, focus ring |
| Destructive | `hsl(var(--destructive))` = `hsl(0 84% 60%)` / approx. `#ef4444` | Terminal connection loss, blocked permissions, muted/off media indicators, end-call control only |

Accent reserved for: `Join Meeting`, active connection-status toggle, participant count badge, local avatar, connected/healthy diagnostics, focus rings, and existing primary UI primitives. Do not use accent for terminal warnings, failed connection states, TURN/WebSocket diagnostics, or destructive actions.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `Join Meeting` for the join dialog. Terminal disconnect recovery primary CTA remains `Reload Page`. |
| Empty state heading | `Waiting for another participant` |
| Empty state body | `Share the meeting link to invite someone. Your camera and microphone stay connected while you wait.` |
| Error state | `Your connection to the meeting was lost. Reload the page to reconnect, or leave the meeting and join again.` |
| Destructive confirmation | `End call` / `Leave Meeting`: no additional confirmation modal in Phase 1; the existing destructive control immediately leaves the room. |

Additional copy constraints:
- For terminal peer connection `failed` and unexpected WebSocket close/reconnect exhaustion, show the existing blocking `ConnectionErrorDialog` rather than a toast or silent log.
- Error copy must be plain and recovery-focused. Do not expose SDP, TURN credentials, environment values, tokens, stack traces, container names, or raw WebSocket payloads in UI copy.
- Transient `disconnected` states may appear in the debug/status panel but must not block the participant unless recovery is exhausted or the state becomes terminal.

## Interaction Contract

| Surface | Required Behavior |
|---------|-------------------|
| Join dialog | Preserve existing preview, device selection, name validation, permission-blocked states, and `Join Meeting` CTA. Joining with audio/video enabled is the Phase 1 baseline path. |
| Waiting state | First participant stays in the room with local media visible and participant count `1`; no error dialog appears while waiting for a second participant. |
| Two-participant grid | When participant two joins, both participants remain visible in the existing grid; each remote tile attaches the correct remote stream and keeps audio playable through the mounted video element. |
| Participant focus | Existing tile click-to-focus behavior remains unchanged. Do not add new meeting navigation patterns in this phase. |
| Debug/status panel | Existing status toggle remains the only diagnostic UI surface. If extended, keep values short, bounded, secret-free, and readable at `12px` monospace. |
| Terminal disconnect | Peer connection `failed` and unexpected WebSocket close/reconnect exhaustion open the existing blocking connection error dialog. Primary recovery is `Reload Page`; secondary action is `Leave Meeting`. |
| Transient disconnect | Recoverable browser `disconnected` states may be shown in diagnostics but should not immediately open a blocking dialog. |
| Media toggles | Existing mute/camera controls retain current destructive styling for off states. Toggle resilience beyond the baseline is Phase 3 unless required by the root cause. |
| Third participant | Any third-participant observation is non-blocking Phase 2 signal; do not add Phase 1 UI states or layout rules specifically for 3+ stability. |
